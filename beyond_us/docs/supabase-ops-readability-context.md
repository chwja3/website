# Supabase 운영 가독성 정리 컨텍스트

2026-05-18. Supabase Table Editor에서 `id`, `profile_id`만 보이면 운영자가 행을 한눈에 해석하기 어렵다. 원본 테이블에 `login_id`를 복제하면 닉네임 변경, 비활성화, 복구 때 동기화 지점이 늘어나므로 원본 구조는 유지하고 `ops_*` 운영용 view에서 프로필 정보를 join해 보여주기로 했다.

2026-05-18. Supabase Table Editor 자체에 모든 테이블의 기본 정렬을 프로젝트 단위로 강제하는 설정은 없다. 대신 운영자가 주로 보는 view에 `ORDER BY`를 넣어 최신 행이 먼저 보이도록 한다. 원본 테이블은 앱과 RPC의 정본으로 두고, 운영 확인은 `ops_*` view를 우선 사용한다.

2026-05-18. admin의 `시스템 상태`는 별도 메뉴로 남기기보다 `Events 관리` 안에 붙이는 편이 운영 흐름에 맞다. 같은 화면에서 카드 이벤트 생성, Supabase 파생 상태 재계산, 상태 점검을 이어서 할 수 있고, 과거 Sheet 전환용 dry-run/apply UI는 Supabase 전환 이후 의미가 없어 제거한다.
