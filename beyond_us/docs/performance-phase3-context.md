# 성능 최적화 3차 Context

## 관찰

- 대시보드는 admin 진입과 새로고침 때 `Events`, `Users`, `MissionDefinitions` 기반 집계를 수행한다.
- 추첨권 화면과 앱 가입자 화면은 `RaffleTickets` 전체 행을 읽어 요약을 만든다.
- 사용자 탭은 계정별 캐시가 들어갔지만, 첫 진입과 백그라운드 예열 시 서버 요청이 겹칠 수 있다.
- admin 목록은 데이터가 늘어날수록 DOM 생성량이 커진다.

## 결정

- 이번 단계에서는 새 영구 시트를 만들지 않고, Apps Script CacheService와 프론트 메모리/로컬 캐시를 우선 활용한다.
- 통계 정합성이 중요한 쓰기 작업 뒤에는 관련 캐시를 명시적으로 삭제한다.
- admin 목록은 검색 전에는 일부만 보여주고, 검색 또는 자세히 보기에서 전체를 보여준다.

## 구현 메모

- 대시보드 캐시는 `dashboard_v3`로 올리고 TTL을 180초로 늘렸다. admin 새로고침 버튼은 `force=1`로 즉시 재계산한다.
- 일반 유저 파생 캐시 삭제는 더 이상 대시보드 캐시까지 지우지 않는다. 미션 제출이 몰려도 dashboard 전체 재계산이 매번 발생하지 않게 하기 위해서다.
- 추첨권 활성 번호 요약은 `raffle_ticket_summary_v1`로 90초 캐시한다.
- 추첨권 발급, 회수, 발급 제외 변경 시 추첨권 요약 캐시를 삭제한다.
- `adminGetRaffleAttendance()`는 추첨권 전체 행을 매번 다시 요약하지 않고 요약 캐시를 사용한다.
- `getUserStatus()` 전체 조회는 수련회 참석 시트 읽기를 제거하고, 추첨권 번호 요약 캐시를 사용한다.
- admin 앱 가입자 목록과 추첨권 번호 목록은 검색 전 기본 80개만 렌더링한다. 검색 시 전체 로드된 목록에서 필터링한다.
