# 숙소 배정표 import 컨텍스트

## 입력 파일

- 원본 파일은 `C:/Users/jkjk9/OneDrive/Documents/카카오톡 받은 파일/숙소배정안내.pdf`다.
- 작업 중 한글 경로 인코딩 문제를 피하기 위해 `tmp/pdfs/lodging.pdf`로 복사해 분석했다.
- PDF는 3페이지이며 표 추출 결과 총 217명이다.

## 데이터 해석

- PDF의 `*` 표시는 해당 호수의 방장이라는 안내가 있다.
- `호수` 값에 `*`가 포함되면 `lodging_note`에 `방장`을 넣는다.
- `303*/203*`처럼 여러 호수가 함께 표시된 경우 `*`를 제거한 전체 값을 방 번호로 유지한다.
- PDF의 교구 표기는 `1교구`, `2교구`, `3교구`, `4교구`다.
- 앱 프로필 교구는 `1청`, `2청`, `3청`, `4청` 등이므로 import SQL에서 같은 값으로 정규화한다.

## 매칭 정책

- 활성 프로필 중 이름과 교구가 모두 맞는 후보가 1명일 때만 자동 매칭한다.
- 후보가 없으면 `profile_id`와 `login_id`를 비운다.
- 후보가 여러 명이면 자동 매칭하지 않고 `raw_note`에 `이름 중복 확인 필요`와 후보 닉네임을 남긴다.

## 적용 순서

- `20260619000300_logistics_assignments.sql`이 먼저 적용되어 있어야 한다.
- 그 다음 `20260620000100_lodging_assignments_import.sql`을 실행한다.
- 적용 후 `retreat_logistics_assignments`에서 `source_batch = 'lodging_20260620'`인 row가 217개인지 확인한다.
- `match_status`별 개수를 확인해 자동 매칭, 미가입, 중복 확인 필요 인원을 점검한다.
